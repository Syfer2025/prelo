O projeto Prelo nasceu da necessidade de um motor de diagramação extremamente robusto no navegador. Ao contrário de editores comuns baseados em HTML/CSS fluido, o Prelo trabalha sob o princípio de páginas de tamanho fixo com posicionamento absoluto e renderização vetorial própria (estilo InDesign/Figma). O objetivo técnico é que o layout físico calculado pelo motor alimente tanto o preview quanto o PDF final, sem depender do DOM como fonte da verdade.

Marco Revisado: Fundação TypeScript Revalidada

O protótipo foi estabilizado para compilar com TypeScript estrito, passar lint e iniciar uma suíte de testes automatizados do motor. A base atual valida fluxo entre frames, overflow, quebra greedy sem espaços finais, contrato real do tex-linebreak, caminho KP sem fallback silencioso, preservação de espaços no refluxo sem quebrar fragmentos hifenizados, flowStory(document, storyId) com cadeia de frames por ID, conversão física pt/mm/pol, geometria de página derivada de perfil de impressão em pontos PostScript, caixas físicas de PDF, resolução de frames do flowStory() a partir da geometria física (rectOnTrim) com layout estável ao alterar o bleed, coordenadas físicas absolutas por frame na saída (rectOnPage = TrimBox + rectOnTrim), leading vindo de ParagraphStyle.lineHeight (multiplicador) no Frame Filler, e margem mínima KDP com bleed. Recursos declarados apenas no modelo continuam marcados como arquitetura, não como implementação final.

1O Princípio Central: Story ≠ Frame

A arquitetura tipográfica herdada dos grandes DTPs (como InDesign e QuarkXPress) baseia-se na separação estrita entre conteúdo e geometria:

  • Story (Fluxo de Conteúdo): O texto contínuo, estruturado em parágrafos e spans com estilos aplicados. A Story não sabe onde reside fisicamente; ela é apenas um fluxo de dados com metadados tipográficos.
  • Frame (Caixa de Exibição): Um polígono físico posicionado em uma página (x, y, largura, altura, rotação). Os frames funcionam como janelas que "capturam" e revelam partes da Story.
  • Cadeia de Encadeamento (Frame Chain): Os frames de texto podem ser encadeados. Quando o texto de uma Story enche o Frame 1, a máquina de estados do motor calcula o ponto de corte geométrico exato e envia o texto remanescente para o Frame 2, depois para o Frame 3, propagando-se por diferentes páginas.

2O Pipeline de Composição (Engine)

O fluxo de processamento de texto do Prelo é dividido em etapas unidirecionais:

1. Tokenizer Separa palavras, espaços, quebras e preserva estilos inline.
→
2. Shaper Mede avanços via função injetada; fonte binária real ainda é etapa seguinte.
→
3. Line Breaker Calcula quebras greedy ou KP; a distribuição visual dos espaços ainda falta.
→
4. Frame Filler State Machine posiciona linhas e faz refluxo entre frames.
→
5. Canvas Preview Renderiza coordenadas absolutas na tela; PDF físico é fase separada.

Como Funciona Hoje (Fase 0R/Fase 1 inicial)

O motor ainda aceita texto bruto via flowText(), mas agora também possui a primeira API de documento: flowStory(document, storyId). Essa API resolve a Story, aplica estilos de parágrafo e respeita a ordem declarada em Story.frameChainIds. A camada print-units.ts deriva trim, bleed e margens mínimas em pontos PostScript, e physical-geometry.ts normaliza o documento em páginas com MediaBox/BleedBox/TrimBox e frames relativos ao trim. O flowStory() já consome essa geometria: os frames do layout são resolvidos por framesFromPhysicalGeometry() a partir de rectOnTrim (pontos, relativo ao trim), de modo que o bleed não desloca o texto. A saída do layout já inclui o posicionamento físico absoluto de cada frame (rectOnPage = TrimBox + rectOnTrim, via frameRectOnPage()) e o leading de cada linha vem do ParagraphStyle.lineHeight (multiplicador). O Frame Filler também honra spaceBefore/spaceAfter, o alinhamento left/center/right/justify e o indent de primeira linha, além de quebrar por caractere palavras maiores que o frame. Cada linha já emite runs posicionados ({ text, x, width, style }); a justificação estica os espaços (menos na última linha) e o canvas desenha run a run. Próximo grande salto: exportar PDF (RGB) a partir desses runs + caixas físicas. Ainda faltam o PDF em si, CMYK/PDF-X (server-side), text-wrap de imagem e o editor visual.

Evolução (Fase 1)

O Shaper passará a extrair métricas diretamente dos arquivos binários de fontes TTF/OTF via opentype.js, com critério explícito para migrar a HarfBuzz/rustybuzz se a cobertura tipográfica não for suficiente. O tex-linebreak já está ligado ao caminho KP para calcular breakpoints; a próxima lacuna é transformar essas quebras em runs posicionados e aplicar distribuição real de espaços na renderização. O hypher continuará responsável por pontos de hifenização específicos para português brasileiro (pt-BR).

7Decisões Críticas de Engenharia

1. Desacoplamento do DOM

O navegador é inerentemente fluido. Usar elementos DOM ou contenteditable para cálculo de páginas físicas causa desalinhamentos severos e comportamentos inconsistentes em diferentes sistemas operacionais. A engine do Prelo calcula o layout de forma puramente geométrica, determinando coordenadas (X, Y) absolutas para cada caractere. O HTML Canvas é usado apenas como rasterizador rápido do preview no editor, e os mesmos dados calculados alimentam as primitivas de vetor no PDF final.

2. Knuth-Plass por Segmento

O algoritmo de Knuth-Plass (KP) avalia o parágrafo inteiro para equilibrar os pontos de quebra e, depois, permitir linhas justificadas. Contudo, em livros com frames vinculados, a largura do frame pode mudar de um frame para o outro. Para contornar essa restrição, o Prelo adota a quebra de Knuth-Plass por segmento: otimiza o texto dentro do limite do frame corrente; quando ocorre o transbordo para o frame seguinte, o restante do texto do parágrafo é re-otimizado dinamicamente de acordo com a nova largura do frame receptor. A justificação visual completa só fica correta quando o renderer passar a desenhar runs posicionados com stretch/shrink por espaço.

3. Conversão CMYK na VPS

O padrão web é restrito ao espaço de cor sRGB. A geração de arquivos CMYK para impressão no padrão PDF/X-1a exige a conversão das cores e das imagens utilizando perfis ICC e calibração de limite de tinta (TAC). Como os navegadores não possuem suporte a CMYK nativo, o Prelo gera o layout vetorial estruturado e imagens de alta resolução no cliente e executa o processamento final de cores em uma VPS server-side rodando utilitários Ghostscript e perfis ICC calibrados.

4. Medição de Fonte Injetada

Na Fase 0, implementamos o Shaper com uma interface que aceita a injeção de uma função de medição externa (MeasureFn). Isso permitiu usar temporariamente a função nativa ctx.measureText() do canvas para acelerar o desenvolvimento do Frame Filler, mantendo o motor totalmente preparado para receber o leitor de arquivos binários opentype.js na Fase 1 sem a necessidade de reescrever a engine.

Knuth-Plass + Multi-frame Verificar se o algoritmo KP comporta larguras de coluna variáveis no mesmo parágrafo. O KP clássico espera larguras uniformes; mudanças físicas de caixa podem quebrar o balanceamento. Calcular o KP individualmente por fragmento de caixa e refazer o cálculo ao trocar de frame.
Métricas opentype.js Garantir que as larguras medidas pelo opentype.js e as calculadas pelo browser sejam compatíveis. Se houver divergência nas larguras de caractere, o cursor desenhado no editor sairá de sincronia com as letras. Utilizar as larguras exatas da fonte obtidas no opentype.js para posicionar cada letra individualmente no canvas.
Text-wrap por Alfa em Tempo Real Validar o tempo de processamento para extrair contornos alfa de imagens grandes durante redimensionamento. Cálculo lento (marching squares) causará travamentos de frames (lag) ao arrastar imagens pela página. Fazer debounce do cálculo de contorno e gerar uma máscara simplificada de baixa fidelidade durante o arrasto.
